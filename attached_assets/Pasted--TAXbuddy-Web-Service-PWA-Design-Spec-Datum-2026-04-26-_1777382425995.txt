# TAXbuddy — Web Service + PWA Design Spec

**Datum:** 2026-04-26  
**Status:** Approved  
**Produkt:** TAXbuddy (ehemals TaxPilot) — KI-Steuer-Assistent für Kleinunternehmer in Deutschland

---

## 1. Kontext & Ausgangslage

### Aktueller Stand
- Einzelne HTML-Datei (~418 KB), React 19, alles inline gebündelt
- Komplett offline via `localStorage`
- Kein Backend, kein Server
- Mobile-optimiert, aber kein echter PWA-Support
- KI-Assistent: offline Wissensbasis + optionaler eigener Anthropic API-Key

### Ziel
Vollständige Umstrukturierung zu einem professionellen Web Service mit:
- Vite + React + TypeScript Frontend (selbes Design, in Komponenten aufgeteilt)
- Node.js + Express + TypeScript Backend
- PostgreSQL Datenbank (EU-Region, DSGVO-konform)
- Progressive Web App (installierbar, offline-fähig, Push-Benachrichtigungen)
- Benutzerkonten + Cross-Device-Sync
- Stripe-Abonnements (Free / Pro)
- Claude API Proxy mit DSGVO-konformer Pseudonymisierung

## 6. Authentifizierung & Sicherheit

### JWT-Strategie
- **Access Token:** 15 Minuten, im Speicher (nicht localStorage) — XSS-resistent
- **Refresh Token:** 30 Tage, httpOnly Cookie (SameSite=Strict) — XSS + CSRF-resistent
- **Refresh Flow:** Transparent, automatisch via Axios-Interceptor
- **Logout:** Refresh Token serverseitig invalidieren (Blacklist in DB)

### Weitere Sicherheitsmaßnahmen
- HTTPS überall (Vercel + Railway)
- Helmet.js (HTTP Security Headers)
- CORS: nur eigene Domain erlaubt
- Rate Limiting: besonders auf `/api/ai/chat` und `/api/auth`
- Passwort-Hashing: bcrypt (12 rounds)
- Zod-Validierung auf allen Eingaben

## 7. DSGVO / Datenschutz

### Grundsätze
- Alle Daten in EU gespeichert (Railway EU West — Amsterdam)
- Keine personenbezogenen Daten in Logs
- Datenminimierung: nur was nötig wird gespeichert

### Claude API (Anthropic)
- **Pseudonymisierung im Backend** vor jedem API-Call:
  - Namen, E-Mails, IBANs werden entfernt
  - Nur Beträge, Kategorien, Steuerfakten werden gesendet
- Benutzer stimmt explizit zu, dass anonymisierte Finanzdaten Anthropic erreichen
- DPA (Auftragsverarbeitungsvertrag) mit Anthropic abschließen

### Stripe
- Stripe ist DSGVO-konform, verarbeitet keine Daten auf unserem Server
- Wir speichern nur: `stripeCustomerId`, `subscriptionStatus`, `subscriptionEnd`
- Kein Kartendaten-Kontakt auf unserem Server

### Pflichtdokumente
- [ ] Datenschutzerklärung (Anthropic + Stripe als Sub-Processors nennen)
- [ ] Impressum
- [ ] AV-Vertrag mit Anthropic
- [ ] Cookie-Hinweis (falls Analytics hinzukommen)

### Nutzerrechte (implementiert)
- `GET /api/user/export` — Auskunftsrecht (Art. 15 DSGVO)
- `DELETE /api/user` — Löschrecht (Art. 17 DSGVO)

## 9. Offline Sync

### Strategie
1. Benutzer macht Änderung offline → sofort in IndexedDB Queue gespeichert
2. UI wird optimistisch aktualisiert (sofortiges Feedback)
3. Service Worker erkennt Verbindung wieder
4. Background Sync sendet Queue an Backend
5. Bei Konflikt → UI benachrichtigt Benutzer

### Cache-Invalidierung
- React Query: `staleTime: 5 Minuten`, `gcTime: 24 Stunden`
- Bei Background Sync: Query-Cache invalidieren

## 11. Error Handling

### Frontend
- React Error Boundary → Fallback UI bei Render-Fehlern
- React Query → automatischer Retry (3x) + stale-while-revalidate
- Offline-Banner wenn Verbindung fehlt
- Toast-Notifications → Fehlermeldungen auf Deutsch

### Backend
- Zod-Fehler → 400 mit klarer Feldangabe
- JWT ungültig → 401
- Pro-Feature ohne Abo → 403
- Rate Limit überschritten → 429 mit Retry-After
- Server-Fehler → 500 mit Error-ID (geloggt, nicht exponiert)
- Globale Error-Middleware — kein uncaught error
